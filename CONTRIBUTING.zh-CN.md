# 贡献指南

感谢你帮助将一个企业级数字助理容器平台从概念变为一个可以实际运行、可审计、可演进的系统。

本仓库仍处于早期公开阶段。目录结构、构建流程和开发工作流可能会继续演进。
如果部分实现、脚手架或内部模块尚未完全开放，请先在 Issue 或 PR 中说明你的意图，我们会尽力保持协作边界的清晰。

## 1. 欢迎贡献的方向

我们特别欢迎以下方向的贡献：

- 容器编排、任务调度与运行时隔离
- 多用户身份感知、租户边界与会话隔离
- 分层记忆、记忆蒸馏、经验升级与组织知识积累
- 技能标准、共享、治理与兼容层
- 可审计性、回放、状态监控、成本治理与风险管控
- 企业 IM、企业微信、Webhook、渠道集成与网关适配
- 文档、测试、示例、部署脚本、可观测性与 SRE 相关工作

## 2. 较大变更前的准备

在开始较大变更之前，请先做以下事项：

1. 搜索已有的 Issue 和 PR，避免重复工作。
2. 如果变更较大，请先开一个讨论性质的 Issue，说明问题、方案和预期影响。
3. 破坏性变更、许可证或商标相关变更、安全边界变更，请在动手实现前先讨论。
4. 如果你计划引入上游项目的代码，请在 Issue 或 PR 中提前标明源仓库、路径和许可证。

## 3. 贡献原则

我们希望贡献遵循以下工程优先级：

1. **隔离优先**
   EnClaws 的核心价值之一是企业级多用户、多任务、多容器运行时。任何可能打破租户、身份或会话边界的设计，必须首先接受审视。
2. **可审计优先**
   影响行为、证据链、责任追溯或成本可见性的变更，应尽可能做到可记录、可回放、可解释。
3. **最小惊讶原则**
   配置、默认值、权限边界和自动化行为，不应表现得像隐藏的舞台机关。
4. **兼容性与迁移说明**
   如果变更影响配置、环境变量、部署、技能接口或消息协议，请附上迁移说明。
5. **文档与实现同步推进**
   如果用户可见的行为发生变化，请同时更新文档。不要让非 README 的文档变成断开连接的灯塔。

## 4. Pull Request 要求

请尽量让每个 PR 专注于单一主题。一个好的 PR 通常应该：

- 说明变更了什么以及为什么
- 当行为发生变化时，包含复现步骤、对比结果、截图或日志
- 如果涉及部署或配置变更，描述迁移和回滚步骤
- 添加或更新必要的测试
- 避免在同一个 PR 中混入无关的重构、纯格式修改或重命名
- 绝不包含密钥、令牌、客户数据、生产环境配置或敏感日志

推荐使用以下提交前缀（非强制）：

- `feat:`
- `fix:`
- `docs:`
- `refactor:`
- `test:`
- `build:`
- `chore:`

## 5. 上游代码与许可证合规

EnClaws 有明确的上游引用和衍生关系，因此贡献者应当诚实地说明来源。

如果你的 PR 复制、改写、移植或大量引用了第三方项目的内容，请：

1. 在 PR 描述中注明源仓库、文件路径、提交哈希或版本号。
2. 确认许可证兼容性，并保留原始版权和许可声明。
3. 对修改过的上游衍生文件，添加清晰的修改说明。
4. 根据需要更新 `THIRD_PARTY_NOTICES.md`、`NOTICE` 和 `LICENSES/third-party/`。
5. 不得移除仍然适用的上游作者署名、版权、商标或归属声明。

## 6. 贡献的默认许可证

除非你在提交时明确声明其他安排且维护者明确接受，否则你提交到本仓库的任何代码、文档或其他受版权保护的贡献，均按仓库的顶层许可证进行贡献。

这也意味着你应确保自己有权提交该贡献，且不会引入未披露的第三方限制。

## 7. 不要公开报告安全问题

如果你发现了**安全漏洞**，请不要提交公开的 Issue。
请改为遵循 `SECURITY.md` 中描述的流程。

## 8. 沟通语言

中文和英文均可。
清晰和完整比语言本身更重要。

## 9. 商标与品牌

如果你的变更涉及项目名称、Logo、包名、对外营销文案、衍生发行版命名或品牌展示，请同时阅读 `TRADEMARKS.md`。
开源代码并不意味着品牌可以随意使用。
